# Отчёт: tier-based стоимость promotion

## 1. Модель цены

**Цена в знаках = tier целевого шаблона** (`PromotionCostFromTargetTier(tier int)` → для tier ≥ 1 возвращается `tier`; при `tier < 1` — 1).

Примеры: повышение в **tier 2** стоит **2**, в **tier 3** — **3**, и т.д.

## 2. Почему так

- Один параметр, без фракций/ролей/линий.
- `unitdata` не раздувается полями цены.
- Ранний цикл **tier 1 → tier 2** остаётся **2** знака — как раньше при фиксированной цене **2**; награда **+1 за победу** не менялась.

## 3. Где вычисляется стоимость

Пакет `internal/game`:

- **`promotionCostForNextPromotion(h)`** — внутренний lookup: `UnitID` → `UpgradeToUnitID` → целевой шаблон → `PromotionCostFromTargetTier(next.Tier)`.  
  Используется в `EvaluatePromotionGate` после успешного `ValidatePromotionDomain`.
- **`PromotionTrainingMarkCostForHero(h)`** — публичный helper для UI: домен + тот же lookup (единая точка для карточки).
- **`PromotionGateResult.Cost`** — поле с вычисленной ценой (для gate, сообщений и списания).

## 4. `hero.TryPromoteHero`

Без изменений: домен по-прежнему не знает о знаках и цене.

## 5. UI

- **Карточка бойца (inspect):** по-прежнему передаётся **актуальная** цена для выбранного героя (`PromotionTrainingMarkCostForHero` в `draw.go`).
- **HUD:** убрана одна фиксированная цифра; вторая строка: **«Знаки обучения: N (цена повышения — в карточке бойца)»**, чтобы не показывать ложную глобальную цену при разных tier.

## 6. Списание

- После успешного `TryPromoteHero`: `g.TrainingMarks -= gate.Cost`, где `gate` вычислен **до** promote (цена соответствует фактическому шагу).

## 8. Ранний loop

- Первое повышение (цель **tier 2**): **2** знака, как при старой фиксированной цене.
- **+1** за победу без изменений.

## 9. Сознательно не делалось

- Новые валюты, награды за бой, баланс-пасс по всему проекту.
- Поля цены в `UnitTemplate`.
- Tier 3+ юниты в контенте только ради теста.

## 10. Следующие шаги (3–5)

1. При появлении tier 3 шаблонов — проверить ощущение цены **3** знака при том же **+1** за бой.
2. При необходимости — мягкий множитель (например `tier+1`) в одной функции.
3. Показать в HUD «минимальную» цену следующего шага лидера — только если понадобится без новой панели.
4. Сохранение `TrainingMarks` в save — отдельный этап.
