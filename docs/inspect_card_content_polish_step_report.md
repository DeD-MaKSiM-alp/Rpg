# Inspect card content polish — step report

## 1. Принцип приоритета информации

Три уровня, отражённые в разметке:

| Уровень | Содержание |
|--------|------------|
| **1** | Заголовок (имя), бейдж ранга/типа боя, контекст, ОЗ, полоса HP, показатели в одной строке с лечением |
| **2** | Профиль (линия · фракция + короткая заметка), способности по одной короткой строке |
| **3** | Развитие: боевой опыт/удар, затем компактные строки повышения; в составе — короткие контекстные примечания (резерв / 0 ОЗ) |

Дублирование имени юнита в секции «Профиль» убрано: имя уже в заголовке.

## 2. Что сокращено по секциям

### Профиль

- Первая строка: **линия · фракция** (без повтора `«DisplayName»`, который дублировал заголовок).
- `InspectNote` обрезается до **72** рун (раньше до 100 / 90).

### Показатели

- Одна строка: **Атака · Защита · Инициатива · Лечение +N** (лечение перенесено из отдельной строки).
- Разделители без лишних пробелов (`·`).

### Способности

- По-прежнему **одна строка на способность**, без префикса `· ` в тексте.
- Короткие подписи: «Базовый удар», «Масс-лечение» и т.д.

### Развитие

- Общая строка опыта: `inspectCombatXPProgressLine` — `Опыт N · удар +X (лид A + опыт B)` (короче формулировка «лид» вместо «лидер»).
- **Повышение** (`inspectPromotionLines`):
  - одна ветка: одна строка про следующий класс, стоимость и контекст (лагерь / знаки / P);
  - несколько веток: **одна** строка-сводка с ценами и маркером `▸` у выбранной ветки; при выбранной ветке — вторая строка со статусом (лагерь / не хватает знаков / готово к P).
- Бой: не более **5** строк в блоке «Развитие» (`maxBattleInspectProgressLines`).
- Состав: не более **6** строк (`maxFormationInspectProgressLines`), с тем же ядром, что в бою; короткие строки «Резерв: …» и «0 ОЗ — …».

## 3. Развитие — что изменилось по смыслу

- Сохранены: опыт, разложение бонуса к удару, ветки повышения, стоимость в знаках, привязка к лагерю.
- Убраны: длинные повторы («Повышение: две ветки…», отдельная строка «нажмите ← или →»), перечисление tier/роли в каждой ветке (достаточно имени и цены).

## 4. Способности — что изменилось

- Только сокращённые подписи и отсутствие декоративного префикса в строке; набор способностей и порядок не менялись.

## 5. Высота карточки

- У союзника убрана **одна** строка за счёт слияния лечения в «Показатели».
- «Развитие» и ветки повышения дают меньше строк при нескольких ветках.
- Вертикальные константы `EstimateInspectCardHeight` и ритм секций не перестраивались намеренно.

## 6. Battle vs formation

- Один и тот же `templateProfileShortLines`, `abilityLinesBullet`, `inspectCombatXPProgressLine`, `inspectPromotionLines`.
- Различия только контекстные: подпись боя / слота / резерва, лимиты строк и хвостовые примечания в составе.

## 7. Изменённые файлы

| Файл | Изменение |

| `internal/ui/battle_inspect.go` | Профиль врага/героя, статы в одну строку, способности, прогресс боя, лимит строк |
| `internal/ui/character_inspect.go` | Состав: статы, прогресс, `inspectCombatXPProgressLine`, новый `inspectPromotionLines` |
| `internal/ui/battle_inspect_test.go` | Проверка строки опыта |
| `internal/ui/inspect_card_test.go` | Тесты бюджета строк, слияния статов, способностей, компактных веток повышения |

## 8. Тесты

- Нет raw `UnitID` в flatten (как раньше).
- Слияние лечения в `StatsLine`, пустой `ExtraStatLine`.
- Верхняя граница числа строк «Развитие» (бой / состав).
- Наличие «Базовый удар» у стартового героя.
- Враг без progression (существующий тест).
- Компактный вывод при двух ветках (`EmpireWarriorSquire` + выбранная ветка).

## 9. Ограничения

- Имена способностей в карточке расходятся с полным именем в `battle/ability.go` (там по-прежнему длинные строки для лога боя) — это осознанно: карточка только для быстрого чтения.
- Длинные `InspectNote` по-прежнему обрезаются, а не переписываются по смыслу.

## 10. Следующие логичные UX-шаги

- При необходимости выровнять подписи способностей в боевом HUD с короткими именами карточки.
- Точечная правка `InspectNote` в данных юнитов, если после обрезки тексты режутся слишком агрессивно.
- Опционально: ещё один проход по enemy-only строкам без шаблона (legacy).
