# Технический аудит проекта (актуальное состояние репозитория)

**Дата среза:** по текущему коду в репозитории (без опоры на старые отчёты как на истину).  
**Цель:** зафиксировать, что реально собрано, где фундамент устойчив, где временные решения, и что логично планировать дальше.

---

## 1. Executive summary

### Текущее состояние проекта

Проект — **работающий прототип early RPG foundation** на Go + Ebiten: бесконечный тайловый мир с чанками, пошаговое движение, контакт с врагом → **полноценный пошаговый бой** → возврат в мир с **синхронизацией HP**, **party-wide XP** на победе, **выбор награды лидеру**, explore-**recovery**, **состав Active/Reserve**, **рекрут** (F9 + world pickup), **inspect** карточки бойца. **Сохранений, инвентаря, квестов и data-driven unit templates по дизайн-докам Империи в коде нет.**

### Что уже собрано хорошо

- **Чёткое разделение слоёв:** каноническое состояние бойцов — `hero.Hero` в `party.Party`; бой — `battle.CombatUnit` из `CombatUnitSeed` (`internal/battle/encounter.go` явно называет conversion layer).
- **Замкнутый контур боя:** старт боя → апдейт → исход → post-battle → (при победе) sync HP → XP → награда лидеру → explore.
- **Party-модель:** Active/Reserve, лимиты слотов боя и ростера, пост-battle sync по `PartyActiveIndex`, recovery на весь ростер.
- **Минимальный world hook под RPG:** типизированный пикап рекрута, отдельный режим подтверждения, маркер сбора.

### 3–5 главных ограничений / точек роста

1. **Дизайн Империи (unit_id, линии, tier) существует в markdown, но не отражён в типах `Hero` / данных рекрута** — рекрут и старт всё ещё «один шаблон героя», не `empire_warrior_recruit` и т.д.
2. **Мир и мета-прогресс:** нет save/load; `BattlesWon` и прогресс сессионные; поражение/отступление почти не трогают мир.
3. **Лидер-центричные награды** после боя сохраняются параллельно party-wide XP — осознанно, но для долгосрочного баланса понадобится политика наград не-лидерам или общий пул.
4. **UI:** функционально закрывает базовые сценарии, но нет портретов, сравнения героев, inspect из боя, богатого feedback для progression.
5. **Тесты:** автотестов по пакетам нет — регрессии ловятся вручную.

## 2. Подробный разбор по разделам

### Battle

| | |
|--|--|
| **Состояние сейчас** | Полный цикл: encounter из мира (один враг), `BuildBattleContextFromEncounter`, пошаговый бой, исходы Victory/Defeat/Retreat, HUD v1/v2 (`BattleHUDStyle`), mouse + keyboard, лог, feedback (`internal/battle/feedback.go` и др.), post-battle overlay. Эскалация врагов от `BattlesWon` (`ScaleEnemyTemplate`). |
| **Что уже хорошо** | Явный `CombatUnitSeed` / `Origin` для связи с миром и партией; формации слотов игрока; разделение preview/resolve. |
| **Проблемы / ограничения** | Encounter **одиночный** (один enemy ID). В шаблоне врага **Bandit** в `GetEnemyTemplate` выставлен `RoleHealer` (`internal/battle/template.go`) — выглядит как несоответствие роли и имени (технический долг). Нет связи врагов с дизайн-доком фракций. |

| **Где в коде** | `internal/battle/*`, `internal/game/startBattle`, `updateBattleMode`, `sync_party.go` |
| **Критичность** | **medium** (для горизонтального контента) |
| **Что блокирует** | Массовые бои, разнообразие encounter’ов «как в RPG» без доработки encounter-слоя. |

### Party / Hero / Reserve

| | |
|--|--|
| **Состояние сейчас** | `party.Party{Active, Reserve}` — единственный канонический ростер. `PlayerCombatSeeds()` только из Active с HP>0; `Leader()` = Active[0]; лимиты `MaxActiveBattleSlots`, `MaxPartyMembers`. Перестановки, reserve↔active, `HeroAtGlobalIndex` для inspect. |
| **Что уже хорошо** | Мало дублирования: один `Hero`, одна проекция в seed; sync после боя по индексу. |
| **Проблемы / ограничения** | Все герои **одинаковы по структуре** — нет `unit_id` / фракции в `Hero`. Рекрут — `NewRecruitHero()` + опциональный `RecruitLabel`, не связан с Empire tier tree. `TwoMemberDemoParty()` в коде есть, **main** стартует с `DefaultParty()` (один герой). |
| **Почему важно** | Без поля шаблона юнита нельзя честно подключить дизайн-доки и ветки прогрессии по `upgrade_to`. |
| **Где в коде** | `internal/party/party.go`, `internal/hero/hero.go`, `internal/hero/recruit.go` |
| **Критичность** | **high** для следующего крупного шага «данные Империи в коде» |
| **Что блокирует** | Разные архетипы, ветки, баланс по линиям из `docs/races_and_npc_roles.md`. |

### Progression / Post-battle

| | |
|--|--|
| **Состояние сейчас** | Победа: `syncPartyFromBattle` → `ApplyVictoryCombatXPForActiveSurvivors` (только живые player units с валидным индексом). Поражение/retreat: только sync HP, без XP. Post-battle: результат → при победе **выбор награды из пула**, `ApplyReward` на **лидера** (`internal/postbattle/postbattle.go`, `progression.ApplyReward`). `CombatExperience` даёт бонус к эффективному basic attack через `EffectiveBasicAttackBonusForCombat()` в seed. |
| **Что уже хорошо** | Два слоя явно разделены в комментариях: party-wide XP vs лидерская награда. Порядок вызовов детерминирован. |
| **Проблемы / ограничения** | Награды по-прежнему **только лидеру** — альтернативы (выбор для каждого актива, общий пул очков) не заложены. Нет «уровня» как сущности — только XP и статы. Прозрачность для игрока: inspect показывает XP и бонус; post-battle не объясняет party-wide XP отдельной строкой (частично компенсировано hint’ом в `postbattle/draw.go` для победы). |
| **Почему важно** | Рост партии и ощущение справедливости наград — риск при расширении состава. |
| **Где в коде** | `internal/game/update.go` (ветка Victory), `internal/progression/*`, `internal/postbattle/*` |
| **Критичность** | **medium** |
| **Что блокирует** | Сложные системы наград и «fair progression» для всей партии без дизайна и кода. |

## 3. Что уже можно считать устойчивым фундаментом

- **Пакетная граница `hero` ↔ `party` ↔ `CombatUnitSeed` ↔ бой** и post-battle **sync по индексу**.
- **Режимы игры** (explore / battle / formation / recruit offer) и **порядок апдейта** боя с post-battle.
- **Party-wide `CombatExperience`** и отдельный **лидерский reward pool** как согласованная модель этапа.
- **World pickup interaction** как расширяемая точка (типы пикапов).
- **Единая тема UI** и разделение оверлеев по режимам.

*Эти части не требуют немедленного архитектурного сноса для типичного следующего фича-слоя (data-driven unit defs).*

## 5. Топ следующих шагов по приоритету

### 1) Слой данных юнита (минимальный) — **приоритет: высокий**

- **Что:** ввести в домен опциональный **`unit_id` / ссылку на шаблон** (или embedded template id) для `Hero`, заготовки из дизайн-дока (хотя бы одна линия Империи).
- **Почему сейчас:** без этого дизайн-доки и код расходятся; блокирует recruit variety и inspect «по канону Империи».
- **Эффект:** единая правда для recruit, inspect, будущей прогрессии.
- **Затрагивает:** `hero`, `party`, recruit flows, позже battle отображение имени/роли.
- **Риски:** переусложнить — держать минимум полей.

### 2) Политика наград / отображение party-wide vs лидер — **средний**

- **Что:** явное UI-резюме после боя («опыт получили: A, B» уже частично в духе; усилить или добавить одну строку в post-battle result step).
- **Почему:** при росте партии игрок должен видеть справедливость системы.
- **Эффект:** меньше скрытой механики.
- **Затрагивает:** `postbattle`, `ui`.

### 3) Расширение recruit sources с данными — **средний**

- **Что:** таблица пулов (хотя бы JSON/YAML в репо) по `early_game_availability` из дока; world pickup выбирает из пула.
- **Почему:** world recruit сейчас один тип бойца; док описывает много юнитов.
- **Эффект:** разнообразие без переписывания мира.
- **Затрагивает:** `world`, `game`, `hero`.

### 4) Тесты регрессии критического контура — **средний**

- **Что:** unit-тесты на `PlayerCombatSeeds`, sync party, progression XP, может pure функции progression/postbattle.
- **Почему:** ручной тест не масштабируется.
- **Риск:** время на настройку.

### 5) Encounter / враги: мульти-враг или хотя бы конфиг encounter — **ниже, пока не нужен горизонтальный контент**

- **Что:** несколько врагов или данные волны.
- **Почему позже:** текущий vertical slice закрыт; усложнение боя без data layer по игроку даст меньше пользы/время.

### Что пока **не трогать** без необходимости

- Полный **save/load** (большой объём, мало пользы без контента).
- **Полный metagame UI overhaul**.
- Глубокий **редизайн боя**, пока не исчерпан слой данных и контента.
