# Отчёт: контекстный inspect по ПКМ (юниты в бою и составе)

## 1. Где работает RMB-inspect

- **Бой (ModeBattle), до post-battle:** ПКМ по карточке юнита в ростере или по токену на поле (тот же hit-test, что и у отрисовки HUD v2) открывает плавающую панель. Повторный ПКМ по **тому же** юниту закрывает панель. ПКМ по **другому** юниту переключает карточку. ПКМ по пустому месту закрывает. **Esc** закрывает; при этом Esc не инициирует отступление в том же кадре (`SuppressEscThisFrame` в `BattleContext`).
- **Состав (F5 / ModeFormation):** ПКМ по строке героя открывает ту же карточку, что и клавиша **I** (переиспользуется `DrawCharacterInspectOverlay`). Повторный ПКМ по **выбранной** строке закрывает карточку; по другой строке — переключает выбранного героя и оставляет карточку открытой.

## 2. Какие юниты поддерживаются

| Контекст | Союзники | Враги |
|----------|----------|--------|
| Бой | Герои отряда: снимок `hero.Hero` с ОЗ из `CombatUnit` + строки из `buildInspectLines` (прогрессия, ветки promotion при двух целях — выбор ←/→ на уровне игры). | `buildEnemyInspectLines`: ОЗ, профиль из шаблона или legacy Def, статы, способности; без progression. |
| Состав | Все строки списка (строй + резерв) через существующий inspect. | Не применимо. |

## 3. UI-паттерн

- Одна **неблокирующая по полноэкрану** панель по центру (`DrawBattleInspectOverlay`), те же приёмы оформления, что и post-battle/character inspect (тема, рамка, типографика).
- Пока открыта карточка в бою: `BattleContext.BlockPlayerInput = true`, чтобы не конфликтовать с выбором способности/цели; **Esc** и **ПКМ** обрабатываются в `game` до `battle.Update`.
- `SuppressMouseRightThisFrame` предотвращает срабатывание боевого «ПКМ = отмена цели» в том же кадре, что и открытие/закрытие inspect.

## 4. Какие данные показываются

- **Союзник в бою:** имя/заголовок как в inspect, подзаголовок «В бою · слот», строки из `buildInspectLines` (профиль, опыт, бонус к базовой атаке, promotion при наличии, способности).
- **Враг:** имя (+ суффикс identity при наличии), «Враг», ОЗ, линия профиля (шаблон или роль/архетип), атака/защита/инициатива, способности; явная строка, что прогрессия к врагам не применяется.

## 5. Что переиспользовано из старой системы

- `buildInspectLines` и `heroSnapshotBattleInspect` в `internal/ui/battle_inspect.go` для союзников.
- `DrawCharacterInspectOverlay` без изменений контракта для состава.
- `PromotionTrainingMarkCostForHeroTarget` / `PromotionTargetUnitIDs` для отображения цен веток в бою (те же данные, что и в составе).
- Hit-test: `ComputeBattleHUDLayout` + `pointHitsUnit` (через `HitTestUnitUnderCursor`).

## 6. Что сознательно не делалось

- Новая оконная система, drag-and-drop, отдельный codex.
- Изменение формул боя, опыта, economy, promotion.
- Полный mouse-only редизайн; клавиатурный бой без изменений, кроме блокировки на время открытой карточки.
- Тяжёлые e2e-тесты с симуляцией мыши.

## 7. Тесты

- `internal/battle/hit_test_test.go` — hit-test по центру rect юнита и промах.
- `internal/ui/formation_hit_test.go` — геометрия первой строки состава vs `FormationHitTestGlobalIndex`.
- `internal/ui/battle_inspect_test.go` — `buildEnemyInspectLines` с валидным `TemplateUnitID`.

**Инженерная заметка:** реализация hit-test перенесена из файла с именем `hit_test.go` в `unit_cursor_hit.go`, потому что в Go любой файл `*_test.go` вне пакета тестов не входит в обычную сборку.

## 8. Логичные следующие шаги (3–5)

1. Опционально: не блокировать ввод полностью при открытой карточке, а пропускать только клики по боевой панели (сложнее, но ближе к «информационной подсказке без паузы»).
2. Подсветка юнита под курсором при зажатом Alt или при hover — без смены LMB flow.
3. Локализация подсказок (если появится i18n).
4. Сжать footer-подсказку в бою, если строка переполняет узкие разрешения.
5. Юнит-тест на чистую функцию «RMB: переключить/закрыть» при вынесении из `handleBattleInspectInput`.
