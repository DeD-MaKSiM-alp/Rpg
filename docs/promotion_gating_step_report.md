# Отчёт: gating promotion (лагерь)

## 1. Правило gating

- **Повышение (P) разрешено только когда игрок стоит на клетке с активным лагерём наёмников** — тем же объектом мира, что и для recruit: `entity.PickupKindRecruitCamp` (лазурный маркер), пикап **не собран**.
- Проверка позиции: `World.PlayerStandsOnActiveRecruitCamp(player.GridX, player.GridY)` — та же логика перебора пикапов в чанке, что и для взаимодействия с лагерём.
- Дополнительной валюты, `BattlesWon` и т.п. **нет** (минимальный шаг).

## 2. Почему так

- Лагерь уже существует в проекте как **world pickup** и **ModeRecruitOffer**; не введены новые сущности «CampManager».
- Правило понятно игроку: «сначала зайди в лазурный лагерь, затем F5 → I → P».
- `TryPromoteHero` остаётся **без знания о карте**.

## 3. Где оценивается доступность

| Слой | Файл | Роль |
|------|------|------|
| Домен | `hero.ValidatePromotionDomain` / `validatePromotionDomain` | Только шаблон: UnitID, путь, цель в реестре |
| Gameplay | `game.EvaluatePromotionGate(h, atCamp)` | Домен + «стоим на лагере» |
| Мир | `world.PlayerStandsOnActiveRecruitCamp` | Факт стояния на несобранном recruit camp |

Вызов: `updateFormationMode` при **P** → `EvaluatePromotionGate` → при `Allowed` вызывается `TryPromoteHero`.

## 4. UI / inspect

- `PromotionUILine(h, atCamp)` — строка в карточке: доменная ошибка / «только в лагере» с id следующего шага / «P — …» на лагере.
- Футер карточки зависит от `atCamp` (короткая подсказка про лазурный маркер).
- `DrawCharacterInspectOverlay` принимает `atCamp` из `Game` (позиция игрока в момент отрисовки, в режиме состава совпадает с клеткой до открытия F5).

## 5. Что остаётся в `TryPromoteHero`

- По-прежнему только **применение** promotion при прохождении **доменной** валидации.
- Политика лагеря **не** внутри `hero`.

## 6. Legacy

- Пустой `UnitID`, нет пути, нет цели в реестре — как раньше, через `ValidatePromotionDomain` и тексты `PromotionErrUserMessage`.

## 7. Сознательно не делалось

- Экономика, инвентарь, стоимость promotion.
- Отдельный экран, ветвление upgrade.
- Save/load.

## 8. Следующие шаги (3–5)

1. Списание ресурса / золота на лагере (если появится счётчик).
2. Отдельный UI «лагерь» без привязки к фиксированному chunk pickup.
3. Несколько лагерей / региональных правил.
4. Уведомление на explore HUD, если есть следующий шаг, но игрок не на лагере.
